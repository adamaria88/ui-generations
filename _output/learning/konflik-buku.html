⚔️ Konflik #1 — Convention vs Exploration (Mindset)
Default cara mikir saat doubt: stick conventional atau explore beyond?

📖 Kedua buku bilang apa

📕 REFACTORING UI
"Default itu starting point, BUKAN constraint. Tanya: 'apa yang komponen ini sebenarnya coba capai untuk user?' — jawabannya sering generate format yang lebih baik dari default." — Wathan & Schoger, thinking.md existing

Mindset: EXPLORE. Default = starting point, boleh beyond.

📘 PRACTICAL UI
"Stick conventional patterns kecuali inovasi punya compelling reason. Innovation = risk usability. Jakob's Law: users spend most of their time on OTHER sites." — Dannaway, Bab 7 hal 229

Mindset: SAFE. Convention = default, explore butuh justification.

Rekomendasi gue: A. Convention (Default) 🟢 REKOMENDASI
Alasan:
  • Paper.id = B2B finance. User = operator akuntansi/admin yang udah familiar tools (Xero, Mekari, Jurnal). Convention friendly = onboarding lebih cepat.
  • Aurora DS udah established. Eksperimen pattern baru = risk inconsistency + dev rework.
  • Innovation OK di area peripheral (marketing site, onboarding hero, empty state). Tapi DEFAULT = convention.
  • Pattern selectable cards bisa di-explore di feature spesifik dengan compelling reason (mis. pricing comparison di marketing landing), tapi bukan default untuk semua form.
Komen di sini: pilih A / B / Hybrid + alasan. Atau "setuju rekomen" kalau A oke.
⚔️ Konflik #2 — Borders & Containers
Saat butuh grouping: pakai border atau cukup spacing?

📖 Kedua buku bilang apa

📕 REFACTORING UI
"Border menambah visual weight ke struktur. Struktur tidak butuh perhatian — konten yang butuh. Alternatif: spacing (proximity), background color, shadow tipis. Pakai border HANYA kalau tiga alternatif itu tidak cukup." — Wathan & Schoger, fewer-borders-alternatives

Mindset: SKEPTICAL. Border last resort.

📘 PRACTICAL UI
"Container = strongest visual cue untuk grouping. Cards & dialog boxes common type. Use container ketika grouping tidak clear without it." — Dannaway, Bab 4 hal 106

Mindset: PRAGMATIC. Container OK kalau bantu grouping.

C. HYBRID Context-dependent 🟢

Rekomendasi gue: C. Hybrid (Context-dependent) 🟢 REKOMENDASI
Alasan:
  • Detail page (View Document, profile, settings) = banyak info, density tinggi. Border tiap section = clutter. Spacing aja cukup.
  • Modal/dialog = boundary harus jelas (visual separation dari background). Border bantu.
  • Form panjang dengan multi-section bisa pakai border tipis OR heading + spacing — case-by-case.
  • Aurora `au-dialog` default udah pakai border + radius. Detail page Aurora pattern bisa lean ke no-border kalau ga ada konflik.
Komen di sini: pilih A / B / C + alasan, atau kasih konteks lain yang gue belum cover.
⚔️ Konflik #3 — Pakai Bold Seberapa Sering?
Bold restrictive (only data hero) atau flexible (heading + emphasis)?

📖 Kedua buku bilang apa

📕 REFACTORING UI
"Bold = emphasis. Emphasis hanya bekerja karena langka. Hierarchy lewat warna (dark vs muted) sudah cukup untuk label-value pair. Bold dihemat untuk data yang benar-benar perlu stand out (angka total, judul)." — Wathan & Schoger, thinking.md

Mindset: RESTRICTIVE. Bold langka = bold powerful.

📘 PRACTICAL UI
"Use regular and bold font weights only. Use bold font weight for headings to emphasise them. Use regular font weight for other smaller text." — Dannaway, Bab 5 hal 173

Mindset: FLEXIBLE. Bold untuk heading + emphasis, convention.

C. HYBRID Heading bold, value via color 🟢

PT XYZ Tbk · Jatuh tempo 15 Mei 2026
Detail Pembayaran
Sub TotalRp 5.000.000
PPN 11%Rp 550.000
Grand TotalRp 5.550.000

Heading bold (convention friendly), label muted + value dark regular (Refactoring UI insight), Total bold (hero). Best of both — convention + elegant.

Rekomendasi gue: C. Hybrid 🟢 REKOMENDASI
Rule konkret untuk Paper.id:
  • Bold: Page title, Section heading, Data hero (Total Amount, Grand Total), CTA button text.
  • Regular: Label, value (label-value pair), body text, helper text, secondary info.
  • Hierarchy value pair via color: label text-muted (#718c9e), value text-primary (#133f5d) — bukan via bold.
  • Saat ada doubt: "kalau ini bold, apakah masih ada yang lebih penting yang harus stand out lebih?" Kalau ya, jangan bold.
Komen di sini: pilih A / B / C + alasan.
⚔️ Konflik #4 — Shadow & Elevation
Shadow 2-layer (subtle premium) atau 3-level discrete (sistematis DS)?

📖 Kedua buku bilang apa

📕 REFACTORING UI
Shadow pakai 2 layer (sharp offset + soft ambient). Kombinasi ini bikin shadow terasa "real" — match physical world depth. Avoid harsh single-layer drop shadow. — Wathan & Schoger, shadow-two-layers

Mindset: SUBTLE. Premium feel via 2-layer composition.

📘 PRACTICAL UI
"Creating 3 shadow options (small, medium, and large) works well for most projects. Use small and sharp shadow to slightly raise; larger softer shadow to elevate higher." — Dannaway, Bab 4 hal 130

Mindset: SISTEMATIS. 3-tier elevation token untuk DS.

🎨 Visual — Card Stack di 3 Level Elevation

A. 2-LAYER (Refactoring) Sharp + ambient kombi
Low
Medium
High

Shadow lebih subtle, "earned" feel. Tiap level masih kombinasi 2 layer. Tapi: lebih sulit document sebagai DS token (banyak permutasi).

B. 3-LEVEL (Practical) Small / Medium / Large 🟢
Small
Medium
Large

3 token discrete: shadow-sm, shadow-md, shadow-lg. Lebih sistematis untuk DS, easier handoff dev. Aurora kemungkinan udah punya pattern ini.

Rekomendasi gue: B. 3-level (Practical) 🟢 REKOMENDASI
Alasan:
  • Aurora DS = systematic token. 3-level (small/medium/large) match DS philosophy: predefined options.
  • Refactoring UI 2-layer = better aesthetic, tapi lebih sulit token-ize untuk DS scale.
  • Compromise: pakai 3-level Practical, tapi tiap level internally pakai 2-layer kombinasi (Refactoring UI insight). Hidden complexity.
  • Aurora-OVERRIDES kemungkinan udah lock shadow token. Stick.
Komen di sini: A / B + alasan. Atau "stick Aurora" kalau lo prefer ga keduanya.
⚔️ Konflik #5 — Pilihan Form: Cards vs Radio
Sub-aspect dari #1, tapi specific use-case form selection. (Konsekuensi #1 = decided lewat #1)

📖 Kedua buku bilang apa

📕 REFACTORING UI
"When radio buttons feel limiting, consider selectable cards — bigger target area, more info per option (icon + title + description). Especially for category/plan/profile selection." — Wathan & Schoger, selectable-cards-vs-radio
📘 PRACTICAL UI
"Stick conventional radio buttons. Jakob's Law. Inovasi pattern = risk usability. If ≤10 options, radio buttons safer than dropdown." — Dannaway, Bab 7 hal 235

⚔️ Konflik #6 — Spacing Label-Input Gap
Jarak label ke input: 4px tight (Refactoring) atau 8pt XS (Practical)?

📖 Kedua buku bilang apa

📕 REFACTORING UI
Gap label→input = 4px (sangat dekat, "clearly belongs to"). Gap antar field = 16px (related tapi distinct). Gap antar section = 32px (clearly separate groups). Spacing exponential, bukan linear. — Wathan & Schoger, spacing-system-scale
📘 PRACTICAL UI
8pt grid: XS=8, S=16, M=24, L=32. Label-input pakai XS (8pt). Inner spacing < outer spacing. — Dannaway, Bab 4 hal 137

🎨 Visual — Form Field

A. 4px (Refactoring) Tight, intimately related 🟢
Nama Customer *
↑ gap 4px (tight)

Label clearly "milik" input. Visual binding lebih kuat. Field padat tapi clear.

B. 8pt (Practical) XS spacing scale
Nama Customer *
↑ gap 8px (moderate)

Comfortable readable. 8pt grid consistency dengan token. Tapi: visual binding label-input slightly weaker.

Rekomendasi gue: A. 4px (Refactoring) 🟢 REKOMENDASI
Alasan:
  • 4px = "intimately related" — label dan input adalah 1 unit, bukan 2 element terpisah.
  • Visual rhythm form lebih clean (less spacing redundant).
  • Sudah di-log di IMPROVEMENT-OPPORTUNITIES.md (Form label-input proximity gap) — open item untuk DS Aurora.
  • Worth raise ke Aurora maintainer untuk update .field default gap dari 8px ke 4px.
Komen di sini: A / B + alasan. Atau "stick Aurora 8px" kalau tech effort terlalu mahal sekarang.
