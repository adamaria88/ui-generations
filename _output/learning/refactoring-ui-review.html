5 Framework Utama

Mental model reusable — invoke kapan pun relevan
1
Visual Hierarchy — De-emphasize, Don't Always Emphasize
Invoke: Page terasa "ramai" atau primary element tidak dominant
Hierarchy punya tiga lever: size, weight, dan color. Instinct awal selalu "bikin yang penting lebih besar" — tapi lebih efektif jika secondary element di-de-emphasize. Label bisa jadi abu-abu, teks pendukung lighter weight, meta-info ukuran lebih kecil. Primary element "menang" bukan karena lebih loud, tapi karena kompetitornya lebih quiet.
→ Paper.id: List Pengeluaran — Total (bold, dark) + Nomor Ref (regular, grey) + Tanggal (small, muted). Di Detail card: amount besar bold, catatan/label muted.
Semua text pakai warna yang sama dan ukuran mirip-mirip. User butuh effort ekstra untuk orientasi.
2
Spacing System — Start Big, Then Tighten
Invoke: Layout terasa cramped, atau spacing terasa arbitrary
Default ke lebih banyak whitespace dari yang kamu pikir perlu — lebih mudah dikurangi daripada ditambah setelah struktur terbentuk. Pakai spacing scale berbasis kelipatan 4: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128px. Proximity = relasi: elemen yang berhubungan harus lebih dekat satu sama lain.
→ Paper.id: Label→input gap = 4px (intimately related), field→field = 16px (distinct), section→section = 32px (clearly separate). Sudah di lock ke rules.
Spacing 15px di satu tempat, 13px di tempat lain — arbitrary values tanpa sistem.
3
Color Palette — HSL + Intentional Shades
Invoke: Setup color system, atau warna terasa inkonsisten / random
Gunakan HSL (bukan hex/RGB) — intuitive untuk adjustment. Setiap hue butuh 8-10 shade (100 lightest → 900 darkest). Pure grey tanpa hue hint terlihat lifeless di layar — tambahkan subtle hue tint. Jangan pakai warna sebagai satu-satunya sinyal — selalu pasangkan dengan icon atau text (accessibility).
→ Paper.id: Aurora palette sudah HSL-based. Saat extend untuk custom state: gunakan hue yang sama dengan brand blue, adjust lightness saja.
Random hex values yang tidak relate satu sama lain. Warna biru untuk "approved" tanpa icon — color blind user tidak bisa bedakan.
4
Finishing Touches Checklist
Invoke: UI "functional tapi plain" — siap untuk polish pass
Small targeted additions dramatically raise perceived quality: replace bullets dengan context-specific icons, tambah thin colored accent border di card/nav/alert, custom checkbox/radio dengan brand colors, ilustrasi + strong CTA di empty states. Juga: reduce border clutter dengan mengganti divider ke box-shadow atau background color difference.
→ Paper.id: Custom checkbox sudah implement (.chk class). Icon di action menu sudah implement. Empty state + CTA = next priority.
Shipping dengan browser default form elements. Empty state hanya "Belum ada data" tanpa CTA.
5
Think Outside the Box — Question Component Defaults
Invoke: Komponen terasa boring / generic seperti template
Setiap komponen punya mental default yang sudah terkondisikan. Default itu starting point, bukan constraint. Tanya: "Apa yang komponen ini sebenarnya ingin capai untuk user?" Dropdown navigasi 6 fitur bisa jadi mega-menu 2-kolom. Kolom Name + Role yang tidak perlu sortir sendiri bisa di-stack. Radio plan selection yang penting bisa upgrade ke selectable cards.
→ Paper.id: Table kolom related bisa di-stack (Nama Mitra + Email = 1 kolom). Dropdown action sudah upgraded dengan icons.
Desain setiap komponen persis seperti browser default — tanpa tanya apakah itu cara terbaik untuk use case ini.

#hierarchy #typography #color HIGH VALUE

Labels Secondary, Values Primary
Ch2 · Hierarchy is Everything · p.48
#hierarchy #detail-page HIGH VALUE
Detail card atau info section terasa "flat" — semua teks berat sama, label dan value bersaing visual.
Di pasangan label-value ("Tanggal: 12 Mei 2026"), yang penting bagi user adalah VALUE-nya. Label hanya konteks. Oleh karena itu: value = bold / dark / larger, label = muted / grey / smaller. Label bahkan bisa dihilangkan kalau value sudah self-explanatory.
"The data itself is what matters, the label is just there for clarity."
Invoice detail di Linear: "Due date" (label, 11px grey) di atas "March 15, 2026" (value, 14px dark bold). User langsung scan ke value; label hanya membantu kalau ada ambiguitas.
Detail section dengan 8 baris Label:Value semuanya 14px #333 regular. Atau label di-bold padahal nilai yang dicari user ada di value-nya.
→ Paper.id View Invoice / View Pengeluaran: semua field di detail card (Tanggal, Jatuh Tempo, Nomor, Mitra) — label = muted small, value = dark regular. Untuk data moneter: value bold + larger.

Spacing System — Start Big, Then Tighten
Ch3 · Layout and Spacing · p.66
#spacing #layout HIGH VALUE
Layout terasa cramped, atau spacing antar elemen inkonsisten / arbitrary.
Default ke lebih banyak whitespace dari yang kamu kira perlu. Gunakan spacing scale berbasis kelipatan 4: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128px. Proximity = relasi: elemen yang berhubungan harus lebih dekat satu sama lain. Group-group konten harus punya gap lebih besar antar-group dibanding internal-group.
"A better approach is to start by giving something way too much space, then remove it until you're happy with the result."
Form: label→input = 4px (sangat dekat, clearly related), antar field = 16px (related tapi distinct), antar section = 32px (clearly separate groups). Hierarchy spacing yang jelas tanpa border divider.
Spacing 15px di sini, 13px di sana — arbitrary values tanpa sistem. Atau semua gap sama 8px sehingga group boundaries tidak terasa.
→ Paper.id In-card toolbar: gap antar action = 8px, gap toolbar ke table = 16px, gap antar section form = 24-32px. Scale 4/8/16/24/32 sudah di-lock ke rules.

Action Visual Weight = Semantic Importance
Ch2 · Hierarchy is Everything · p.60
#hierarchy #button #destructive HIGH VALUE
Page terasa "ramai aksi" — beberapa tombol terlihat sama pentingnya, atau destructive action secara visual sama dengan primary action.
Visual weight sebuah action harus match kepentingannya: primary = filled/solid pill (satu per page), secondary = outline/ghost, tertiary = text-link only, destructive = visual berbahaya (merah) tapi weight sesuai frekuensi (jarang = text-link merah, bukan filled merah). Kalau jarang tapi berbahaya = sembunyikan di sub-menu.
"Every action on a page sits somewhere in a pyramid of importance. Most pages only have one true primary action."
GitHub: "Merge PR" = filled green, "Close PR" = outline grey, "Delete branch" = text-link merah kecil. Hierarchy jelas meski tiga action di area yang sama.
Form dengan tiga tombol semua filled (Simpan/Batalkan/Hapus) — semua besar, semua sama weight. Atau destructive di-highlight merah solid besar — justru menarik perhatian ke aksi berbahaya.
→ Paper.id "Tindakan Lainnya" (text-link kiri) + "Catat Pengeluaran" (pill primary kanan). Di Detail: "Ubah" standalone button, "Hapus" masuk sub-menu Tindakan ▾.

Color Palette via HSL — 8-10 Shades per Hue
Ch5 · Working with Color · p.138
#color #accessibility #design-system HIGH VALUE
Warna di desain terasa inkonsisten, palette terasa random, atau warna grey terlihat "lifeless".
Gunakan HSL (Hue/Saturation/Lightness) — intuitive untuk adjustment. Setiap hue butuh 8-10 shade (100 = sangat muda → 900 = sangat gelap). Pure grey tanpa hue hint terlihat lifeless di layar. Jangan pakai warna sebagai satu-satunya sinyal untuk state — selalu pasangkan dengan icon, weight, atau teks.
"HSB is more common in design software, but browsers only understand HSL — so if you're designing for the web, HSL should be your weapon of choice."
Aurora DS palette: --color-light-brand-* series (100-900) dengan hue consistent. Grey Aurora pakai subtle blue hint — itulah kenapa terasa "bersih" bukan "flat".
Arbitrary hex (#f5f5f5, #e0e0e0) yang tidak punya relasi. Warna biru untuk "approved" tanpa icon centang — color blind user tidak bisa bedakan.
→ Paper.id Saat extend Aurora palette: ambil hue yang sama dengan brand blue, adjust lightness saja. Badge status: icon + color (bukan color only).

Line Height by Context
Ch4 · Designing Text · p.122
#typography #readability HIGH VALUE
Headline terasa terlalu loose, atau body paragraph terasa cramped.
Line height bukan satu nilai untuk semua — semakin panjang baris dan semakin kecil font, semakin besar line-height yang dibutuhkan. Headline (large, sedikit kata): 1.0–1.2 (tight). Body text: 1.5–1.8 (loose untuk readability). Small UI labels: 1.2–1.4. Nilai 1.5 sebagai universal default tidak tepat.
H1 "Buat Invoice": font-size: 24px; line-height: 1.2 — tight, authority. Paragraph deskripsi: font-size: 14px; line-height: 1.6 — loose, readable. Table cell: line-height: 1.4.
Semua text di page pakai line-height: 1.5 tanpa pertimbangan — heading jadi aneh spaced.
→ Paper.id Page title: 1.2–1.3 · Table cell: 1.4 · Form helper text: 1.6 · Toast message: 1.4. Aurora sudah set ini di component — ikuti kecuali ada override eksplisit.

Two-Layer Box Shadow for Realistic Depth
Ch6 · Depth · p.186
#depth #shadow #visual-polish MED VALUE
Card atau floating element terasa "flat" atau shadow terlihat artificial.
Shadow di dunia nyata terdiri dari dua komponen: (1) direct shadow — kecil, sharp, dekat sumber cahaya, dan (2) ambient shadow — besar, soft, menyebar. Kombinasi keduanya = shadow natural. Single shadow saja terasa artificial.
Card: 0 1px 3px rgba(0,0,0,0.12), 0 1px 2px rgba(0,0,0,0.24). Dropdown: 0 4px 6px rgba(0,0,0,0.07), 0 10px 15px rgba(0,0,0,0.1).
box-shadow: 0px 4px 4px rgba(0,0,0,0.25) — single symmetric shadow yang terlihat flat. Shadow dengan opacity >0.5 yang terlihat harsh.
→ Paper.id Card summary: 0 2px 8px hsla(0,0%,0%,.08). Dropdown (portal): 0 4px 6px + 0 10px 20px. Modal: cek Aurora au-dialog.component.scss dulu.

Empty State as First Impression
Ch8 · Finishing Touches · p.234
#empty-state #onboarding #visual-polish HIGH VALUE
User buka fitur baru pertama kali dan lihat "Belum ada data" atau table kosong tanpa arahan.
Empty state adalah PERTAMA yang user lihat saat coba fitur baru — bukan afterthought, tapi priority design. Harus punya: (1) ilustrasi / visual yang menarik, (2) CTA button yang jelas, (3) HIDE UI controls yang irrelevant saat kosong (tab, filter, kolom sort — kalau data kosong, kontrol ini misleading).
"Use them as an opportunity to be interesting and exciting — don't settle for plain and boring."
Notion empty database: ilustrasi kecil + "Add a row to get started" + tombol "+ New" primary besar. Tab dan filter disembunyikan. User langsung tau action pertama mereka.
Table kosong dengan header kolom lengkap, tab filter aktif, pagination "0 dari 0 entri" — semua kontrol irrelevant tetap terlihat. User bingung, tidak ada direction.
→ Paper.id Semua List Page (Pengeluaran, Invoice, Customer): saat zero data, hide in-card toolbar + filter row, tampilkan ilustrasi + teks motivasi + tombol "Catat Pengeluaran Pertama".

Fewer Borders — 3 Alternatives
Ch8 · Finishing Touches · p.238
#visual-polish #spacing #depth HIGH VALUE
Layout terasa "busy" atau "cluttered" karena terlalu banyak garis border sebagai separator.
Border adalah cara paling obvious untuk memisahkan dua elemen — tapi bukan satu-satunya, dan terlalu banyak bikin design ramai. Tiga alternatif: (1) Box shadow — lebih subtle, tidak "cutting". (2) Two different background colors — adjacent element pakai bg sedikit berbeda. (3) Extra spacing — spacing sendiri cukup untuk menciptakan separasi.
"Borders are a great way to distinguish two elements, but they aren't the only way, and using too many can make your design feel busy and cluttered."
Contacts list: search field dipisahkan dari list bukan dengan border — pakai background: hsl(200,10%,94%) di search area vs white di list. Separasi jelas tanpa satu garis border pun.
Setiap row table punya border-bottom, tiap section punya border atas dan bawah, card punya border semua sisi + ada divider internal. Design terlihat seperti spreadsheet.
→ Paper.id Row separator: pakai spacing atau subtle bg alternating, bukan border-bottom tebal. Detail card: gunakan bg color difference untuk separasi antar block. Form section divider: extra spacing 32px cukup tanpa garis <hr>.

Accent Border as Quick Visual Polish
Ch8 · Finishing Touches · p.224
#visual-polish #color HIGH VALUE
Card, alert, atau nav item terasa plain dan "belum selesai" meski kontennya sudah lengkap.
Thin colorful border strip (2-4px) di posisi strategis adalah cara termudah menambah visual interest tanpa graphic design skill tinggi. Posisi efektif: (1) top border di card — border-top: 4px solid brand-color, (2) bottom border di active nav, (3) left border di alert/banner. Satu accent cukup — jangan di setiap elemen.
Stripe Dashboard card: thin teal top-border di revenue summary card. Langsung terasa "premium" tanpa ilustrasi. Nav item aktif: brand blue bottom-border 2px — lebih elegan dari full background highlight.
Accent border di setiap card, setiap alert, setiap section — jadi noise, bukan highlight. Pakai warna yang tidak ada di palette.
→ Paper.id Dashboard card summary: border-top 3px solid brand. Alert/banner: border-left 4px solid status-color. Active nav: accent line bawah. Cek Aurora alert/banner component dulu — mungkin sudah punya built-in.

Selectable Cards vs Radio — Upgrade for Important Choices
Ch8 · Finishing Touches · p.245
#form #component #visual-polish HIGH VALUE
Form punya pilihan penting (paket harga, tipe dokumen, mode bayar) yang tampil sebagai radio button plain.
Radio button default cocok untuk pilihan teknis sederhana. Tapi untuk pilihan yang PENTING bagi user, radio circle terasa tidak match dengan bobot keputusannya. Upgrade ke selectable card: card dengan border, konten lebih kaya (nama + harga + deskripsi), dan highlight border/bg saat selected.
"If a set of radio buttons are an important part of the UI you're designing, try something like selectable cards instead."
Change Plan UI upgrade: 4 kartu horizontal masing-masing dengan tier name, GB, price/mo, border-teal saat selected + checkmark icon. User bisa compare semua opsi dengan mudah.
Memilih paket harga Rp 500rb/bulan via radio circle 16px kecil di sebelah teks plain "Business Plan". Decision besar dengan affordance kecil = disconnect.
→ Paper.id ⚠️ Aurora belum punya au-selectable-card. Kalau implement, wajib custom + catat di AURORA-OVERRIDES.md. Hanya untuk pilihan truly significant. Pilihan sederhana ≤4 opsi = tetap Aurora radio.

Text Contrast Over Images
Ch7 · Working with Images · p.202
#imagery #accessibility
Teks di atas foto atau background image sulit dibaca — terutama kalau foto punya area terang tak terduga.
4 teknik guarantee teks terbaca di atas image: (1) Semi-transparent overlay dark rgba(0,0,0,0.4), (2) Low-brightness image, (3) Color tint overlay (brand semi-transparent), (4) Text shadow tipis. Jangan rely pada "gambarnya kebetulan dark" — selalu gunakan salah satu teknik untuk kontrol predictable.
Hero banner: linear-gradient(rgba(0,0,0,0), rgba(0,0,0,0.5)) dari atas ke bawah — area bawah (tempat teks) jadi gelap predictable. Teks putih di atas gradien hitam = selalu readable.
Teks putih langsung di atas foto tanpa overlay, berharap foto cukup gelap — kadang readable, kadang tidak.
→ Paper.id Relevan untuk avatar/banner upload di profile bisnis. Untuk user-generated image: selalu object-fit: cover + background-color fallback + teks di area aman.

Konflik — Perlu Review Lo

5 keputusan yang gue buat sendiri. Konfirmasi atau koreksi masing-masing.
⚠️ Semua ini gue putuskan tanpa lo. Kalau ada yang salah, klik "Koreksi" dan ketik keputusan yang benar. Setelah lo approve semua 5, status buku berubah dari draft → reviewed.
A
Selectable Cards vs Radio
Kapan upgrade radio → selectable card?
📚 Buku bilang Kalau radio buttons adalah bagian UI yang PENTING, upgrade ke selectable cards — card dengan border highlight, konten lebih kaya.
✅ Keputusan gue Tetap Aurora au-radio untuk ≤4 opsi. Selectable card boleh, tapi harus custom + catat di AURORA-OVERRIDES.md. Hanya untuk pilihan truly significant.
Aurora belum punya au-selectable-card component. Custom diam-diam tanpa dokumentasi = pelanggaran aurora-lookup-ritual. Kalau mau, harus eksplisit.
✓ Dikonfirmasi
B
Convention vs Exploration
Seberapa jauh boleh eksplor dari default component?
📚 Buku bilang Default design bisa dieksplorasi jauh — jangan biarkan "existing beliefs" menahan design. Exploration seringkali membawa ke level lebih tinggi.
✅ Keputusan gue Exploration OK untuk form ≤5 field, atau context marketing/onboarding. Form operasional >5 field → tetap convention. B2B finance user butuh familiar patterns.
Practical UI (buku ke-2) push Jakob's Law — stick convention kecuali ada compelling reason. Hybrid: default convention untuk operasional, exploration untuk context yang butuh visual appeal.
✓ Dikonfirmasi
C
Borders — Skeptis vs Pragmatis
Kapan border OK, kapan harus cari alternatif?
📚 Buku bilang Skeptis dengan border. Coba spacing/proximity/bg color dulu. Border = last resort — terlalu banyak border bikin design busy dan cluttered.
✅ Keputusan gue Detail page = skeptis (no border, pakai spacing/bg). Modal/dialog = pragmatis (border boundary harus jelas). Card di list page = pragmatis (container bantu separation).
Detail page density tinggi, border bikin clutter. Modal boundary harus jelas dari background. Context-dependent hybrid lebih practical dari all-or-nothing.
✓ Dikonfirmasi
D
Shadow — 2-Layer vs 3-Level Discrete
Pendekatan shadow: aesthetic vs systematic?
📚 Buku bilang 2-layer composition (direct shadow + ambient shadow) untuk depth yang terasa natural seperti dunia nyata. Setiap card/floating element pakai 2 layer.
✅ Keputusan gue Best of both: 3 token discrete (shadow-sm/md/lg) untuk DS-friendly, tapi tiap token internally pakai 2-layer composition. Token sistematis + depth aesthetic.
Practical UI push 3-level sistematis untuk DS. Refactoring UI push 2-layer composition untuk aesthetic. Dua hal ini tidak conflicting — bisa disatukan.
✓ Dikonfirmasi
E
Ilustrasi di Empty State
Full illustration vs Aurora pattern saja?
